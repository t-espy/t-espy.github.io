Working Notes / Method

A Discipline for Building Software with AI

One of the more interesting questions about AI-assisted development has little to do with how fast you can generate code. It's how you keep a model from confidently building the wrong thing — or shipping buggy code, drifting the architecture, reaching for off-pattern solutions — and how the project remembers why it was built the way it was. These are the conventions I've settled on over a year of shipping with AI assistants as the implementation layer.

00 The premise

I architect; an AI assistant implements. That division works well — until it doesn't, and the ways it fails are specific. A model will assert an API behavior from memory that the vendor's docs contradict. It will re-derive a metric that's already stored. It will lose, between one session and the next, the reason a decision was made, and quietly reopen a settled question. None of these are failures of capability. They're failures of process — and process is the part a human still owns.

What follows isn't a framework or a tool. It's a small set of conventions, each one a response to a failure I actually hit. They're written down in the repo, not held in my head, because the whole point is that they survive a fresh session with a model that has no memory of the last one.

01 Decisions are immutable; the record is append-only

Durable design decisions live in Architecture Decision Records — one numbered file per decision, in Nygard format: context, decision, consequences. The rule that makes them worth keeping is that an accepted ADR is never edited. To change a decision you write a new record that supersedes the old one and flip a single status line. Nothing else changes.

The value isn't documentation for its own sake. It's being able to read what was believed at decision time — including the alternatives that were on the table and the reasons they lost. Editing the record to match today's opinion destroys exactly the thing that makes it useful six months later, when a model (or a person) is about to reopen a question that was already settled for reasons no longer visible in the code.

A companion convention covers plans — the working notes for a task in flight. Plans are freely editable right up until the first commit that references them lands; after that, changes are appended as dated updates, never edited in place. Every plan closes with an outcome line: shipped with commit hashes, abandoned with a reason, or superseded. That single line is what connects a session's work to the durable record instead of letting it evaporate when the session ends.

A guard against the obvious failure mode: ADR theater is worse than no ADRs. The convention names when not to write one — routine choices, bug fixes, style. A record is for decisions that are expensive to reverse, surprising to a future reader, or that rule out an alternative someone would otherwise reach for. When it's unclear, the answer is to ask, not to file.

02 Verify external claims before they harden into documents

This is the rule I'd keep if I could keep only one. Don't assert how an external system behaves from memory when that claim is about to land in a durable document. APIs, framework defaults, broker constraints, library behavior — fetch the published source, cite the URL inline, and only then write it down. Plans seed code; ADRs commit the codebase to a direction. A wrong assumption in either one multiplies into wrong implementation.

This rule didn't come from theory. It came from getting caught.

The incident this rule is named after

A plan to restructure part of the system committed to several confident claims about a third-party framework's behavior — all drawn from the model's memory, none verified. A first verification round found multiple wrong API and behavioral claims, and that the plan had missed the vendor's own primary recommendation for the exact scenario entirely.

A second round, checking the alternative the rewrite had dismissed, found that the dismissed option was in fact endorsed by the vendor's staff with code examples — and surfaced a serious operational hazard the earlier drafts hadn't mentioned. The same recall-from-memory failure caught us twice in one session, on one decision: first on the chosen path, then on the rejected one.

The decision record for that change now documents both rounds. The cost of skipping verification, written into the rule, scales with how load-bearing and durable the document is.

If I'd put it in a pull-request description for a reviewer, it's a decision record. If I'd whisper it to a colleague onboarding the project, it's a memory note.

03 Three kinds of memory, routed on purpose

An AI-assisted project accumulates knowledge in at least three places, and the failure mode is letting them blur. The cross-project assistant memory drifts toward project facts it shouldn't own; design decisions get stranded in a session log nobody reads again. So the routing is explicit:

Belongs in session / assistant memory
Belongs in a decision record
How I want to collaborate — commit small, don't restart services without asking
Design decisions with consequences for the code itself
Who I am and what I'm building, across projects
Reasoning about this codebase a future reader will need
Handoff context — what's in flight, what just shipped
"We chose A over B because…" — including deliberate non-action

The repo's hard-rules file carries the terse what — the standing constraints a session must not break. Each rule that has real reasoning behind it points to the decision record that holds the why. The rule is the guardrail; the record is the argument. Keeping them separate means the guardrail stays scannable and the argument stays available, and neither one rots into the other.

04 The artifacts

Concretely, this is what lives in a repo run under these conventions. None of it is exotic — the discipline is in keeping it current and letting it actually constrain the work, session after session.

  • CLAUDE.md The hard-rules file a fresh session reads first. Standing constraints, terminology, architecture must-knows, and the gotchas already solved — each load-bearing rule cross-linked to its decision record.
  • docs/adr/ Numbered, immutable decision records in Nygard format, with a maintained index and a meta-record (ADR-0001) defining the convention itself — when to write one, and when not to.
  • plans/ Working notes for substantive tasks. Freely edited until first referenced by a commit, append-only after, each closed with a shipped / abandoned / superseded outcome line.
  • tests/ A non-trivial change isn't done until its tests pass. New behavior and bug fixes land with coverage; bug fixes land with a regression test. The suite is the contract a model's output has to satisfy.

05 Where this came from

These conventions were forged on a real, demanding project: a parallel optimization engine for algorithmic-trading research — a stateful, correctness- sensitive system where a wrong assumption doesn't throw an exception, it silently produces a plausible, wrong number. That's the environment that taught me to write the verification rule down. The conventions described here evolved with the later, private version of that engine, which runs well ahead of the public release; the public lean-optimizer page covers the open-source mirror, while this page is about the method that built the private system and several projects beside it.

On scope, plainly: this is a discipline I've built and validated as a solo architect, with AI assistants as the implementation layer — not a process I've run across a team of human contributors, where immutable records and plan handoffs meet different pressures. I think most of it transfers to a team setting, and the decision-record and verification conventions long predate AI. But I'd rather tell you exactly what's been proven than imply more than it has.
