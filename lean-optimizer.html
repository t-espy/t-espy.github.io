QuantConnect LEAN optimization, validation, orchestration, and live-deployment infrastructure

Overview

This work started as a six-month migration of an existing live MultiCharts systematic-trading workflow into a private QuantConnect LEAN research-to-live infrastructure platform. The goal was not simply to port strategy code; it was to build the infrastructure layer around research: repeatable optimization, walk-forward validation, credibility gates, portfolio construction, monitored deployment packages, and controlled live execution.

The public lean-optimizer-public repository exposes the optimizer-engine slice of the private platform. It compiles C# strategies once, runs warm Docker containers with persistent .NET harness processes, and searches parameter space via GA, LHS, Bayesian/Optuna TPE, and local-grid stages. The private system adds IBKR data ingestion, WFO/WFAC credibility validation, portfolio construction, prediction bands, parameter-staleness monitoring, live/paper deployment packaging, dashboard approval, and post-trade analytics.

Measured Optimizer Performance

Public optimizer benchmark against LEAN CLI's built-in lean optimize --strategy "grid search" on a 7-parameter space (679,140 grid points), 15 parallel workers, same strategy, same data:

LEAN CLI Grid Search lean-optimizer (GA)
Search strategy Exhaustive grid (all 679,140 points) Incremental genetic algorithm
Evaluations 679,140 611
Throughput (15 workers) 2.15 evals/sec 3.82 evals/sec
Wall time ~88 hours (extrapolated from 1,772 evals in 825s) 2 min 40 sec
Search efficiency 1x 1,112x fewer evaluations

The two advantages are independent and compound: the persistent harness yields ~1.8x higher throughput per worker, and the GA's directed search converges in 1,112x fewer evaluations. Combined: what takes LEAN CLI's grid search an estimated ~88 hours completes in under 3 minutes. I do not claim a measured multiplier for the full private WFO/WFAC research pipeline; that is described qualitatively as a move from manual one-off backtests to an automated walk-forward credibility pipeline.

Research-to-Live Platform Scope

The private platform spans the full research-to-live loop:

  1. IBKR data ingestion and backfill
  2. C# LEAN strategy packaging and Python strategy ports
  3. Parameter-space definition through YAML research plans
  4. GA, grid, and Bayesian parameter optimization
  5. Out-of-sample evaluation on held-out windows
  6. WFO/WFAC credibility scoring
  7. DEPLOYABLE / MARGINAL / BLOCKED verdicts
  8. Portfolio construction and sleeve sizing
  9. Forward prediction bands and drawdown kill-lines
  10. Parameter-staleness monitoring
  11. Dashboard review and manually approved deployment packages
  12. Post-trade analytics from the IBKR fill ledger

Validation Discipline

The platform is intentionally conservative. It evaluated 7 strategy families across 257 symbols with WFAC verdicts, running 5,563 test plans and 36,254 WFO windows. Only two strategy families — Hull MA trend following and Compression Breakout — passed the credibility gates and were promoted to live/paper deployment. Rotation and mean-reversion candidates were evaluated and rejected by the validation framework. That rejection behavior is part of the system's purpose: preventing attractive but fragile backtests from reaching live trading.

WFAC aggregates walk-forward out-of-sample performance into DEPLOYABLE / MARGINAL / BLOCKED verdicts. Deflated Sharpe Ratio, CPCV, False Strategy Theorem checks, bootstrap confidence intervals, and effective-N corrections are used as diagnostics and guardrails. Live behavior is compared against a pre-committed forward expected distribution rather than against inflated in-regime backtest headlines.

Research Orchestrator

The platform includes a service-based Research Orchestrator outside the main web UI. It can run the research chain unattended for days: data pre-flight, symbol download, Discover, parameter-range validation, WFO bake-off, WFO-to-WFAC promotion, composition, and packaged backtest. The orchestrator interleaves IBKR-throttled minute downloads with otherwise-idle research compute, dispatching each symbol's research once its data is ready.

A bounded Research Steward runs as a separate LLM-guided layer. A deterministic queue runner keeps backtests running from a curated backlog; the LLM steward periodically reads results, writes findings, and queues the next experiments. Its trust boundary is deliberately narrow: it can read and call a restricted steward CLI, but has no raw shell, write, git, database, or live-trading access.

Research orchestration is distinct from live trading. Research can run unattended; real-money deployment remains gated by dashboard review and manual operator approval.

Persistent Harness

The public optimizer's core innovation is the persistent .NET harness. Each Docker container runs a long-lived dotnet /Harness/LeanHarness.dll process that reads JSON requests from stdin, runs backtests with full LEAN state reset between runs, and writes JSON responses to stdout. This eliminates per-eval .NET startup, assembly loading, and LEAN engine initialization overhead.

The harness uses LEAN's regression test reset sequence: Config.Reset(), Composer.Instance.Reset(), SymbolCache.Clear() — the same sequence LEAN's own test suite uses to run multiple backtests in one process. Console.Out is redirected to stderr so LEAN trace/debug output does not corrupt the JSON protocol.

Engineering Controls

  • Tests: 2,168 private test functions across 165 files; 130 tests in the public optimizer subset.
  • ADRs: 39 architecture decision records document durable design choices.
  • AI-assisted development: Claude Code accelerated implementation, but architecture, validation methodology, deployment controls, tests, and operational ownership remained human-owned.
  • Manual deployment gate: real-money deployment is never autonomous.
  • Bounded LLM access: the research steward cannot write code, modify the database, use git, or access live-trading controls directly.

Background

The genetic algorithm at the core of lean-optimizer is the same class of algorithm I developed at Togai InfraLogic in the early 1990s, where my research in fuzzy logic and genetic algorithms was published at IEEE and NASA conferences. Three decades later, the same optimization principles run inside a modern LEAN-based research infrastructure stack.
